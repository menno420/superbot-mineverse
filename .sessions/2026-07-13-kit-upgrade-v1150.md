## 💡 Session idea

`regenerated` per kit-owned CI file — it never records the pre/post
sha256 pair, so "kept (already current)" on this wave versus "regenerated
and STRIPPED the hand-installed guards" on idle's wave (PR #91) are
distinguishable only if the operator did a manual byte-copy ritual before
running the tool. Have the kit's upgrade report log
`<file>: pre=<sha256> post=<sha256> (kept|regenerated)` for every
kit-owned workflow it considers, so customization loss is mechanically
auditable from `.substrate/upgrade-report.md` alone. Dedup-checked: the
games wave's idea is about seat-digest report classes, idle #91's
follow-up is about splitting the host guard into a separate workflow, and
no card/ideas doc here proposes report-level hash pairs.
